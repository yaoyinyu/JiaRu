## 逐指可见性增强 — 三信号融合 + 时序平滑

**问题**: 旧 `isNailVisible()` 仅用 `TIP.z - DIP.z > 0.002` 单信号判定，MediaPipe z 精度不足导致混合态（手偏转时部分指可见）判定不可靠。

**方案**: 三信号融合逐指判定 + 时序平滑防闪烁

| 信号 | 方法 | 阈值 | 优势 |
|------|------|:----:|------|
| A | TIP.z - DIP.z | 0.002 | 传统 z 差（改进后用平滑值） |
| B | TIP.z - PIP.z | 0.003 | 印证 A，PIP 比 DIP 更稳定 |
| C | len2D/len3D 透视缩短比 | 0.65 | 基于 x/y 几何，精度最高 |

**融合策略**:
- 强否定：sigA + sigC 同时否定 → 不可见（保守防误判）
- 多数投票：≥ 2 信号认可 → 可见

**时序平滑**:
- 新增 `visibleSmoothRef[5]` 存储每指可见性平滑值
- EMA(α=0.3) 帧间平滑，阈值 0.5（≈ 2 帧确认）
- 消除单帧噪声导致的渲染闪烁

**代码改动**:
- `isNailVisible()` → `computeFingerVisibility()`：使用平滑后的关键点值
- 判定时机移到 EMA 平滑之后（先平滑关键点，再用平滑值做判定）
- 新增 4 个常量：`NAIL_PALM_Z_THRESHOLD_B`、`FORESHORTEN_THRESHOLD`、`VISIBILITY_EMA_ALPHA`、`VISIBILITY_SMOOTH_THRESHOLD`

## 修复：重新启用全局朝向渲染门控

**问题**: 手心朝镜头时美甲仍会贴在手心侧。`shouldRenderNails()` 用 4 传感器融合正确检测到了手心/手背，但它的 `decision.render` 返回值被忽略，仅用于设置 UI 朝向指示器。唯一阻止渲染的防线是 `isNailVisible()` 的 `TIP.z - DIP.z > 0.002`，但 MediaPipe z 坐标精度不足，手心朝镜头时 z 差值经常不超阈值。

**修复**: 在 `onResults` 回调中添加 `if (decision.render)` 门控：
- **手心**（palmScore >= 2 或叉积+辅助）→ `decision.render = false` → 完全跳过 `paintNails()` 调用
- **手背/侧手/模糊态** → `decision.render = true` → 进入 `paintNails()`，内部由 `isNailVisible()` 做逐指精细过滤

这样形成两层防御：
1. 全局层：4 传感器融合（使用 x/y 高精度信号 + z 辅助），屏蔽所有手心渲染
2. 逐指层：`isNailVisible()` 处理混合态（手偏转时部分指可见/不可见）

## ⚠️ 经验教训

### 1. 阈值设计必须避免覆盖关系
- 两个阈值 `NAIL_VISIBLE=0.002` < `NAIL_AMBIGUOUS=0.004` 产生了包含关系
- 第一个分支 `tipDipDiff < 0.002` 几乎覆盖所有负值情况
- 第二个分支 `|tipDipDiff| < 0.004` 覆盖了 [-0.004, 0.004]
- 两者并集几乎覆盖所有情况 → 逻辑恒为 true
- **规则**：设计多阈值时画出数轴覆盖范围，确保每个分支有独立作用域

### 2. 先实测物理方向，再写代码逻辑
- MediaPipe z 轴朝向镜头为负值（定义，非直觉）
- 手心朝镜头时 TIP.z > DIP.z（指尖比关节更远离镜头）
- 手背朝镜头时 TIP.z < DIP.z 或接近 0（指甲可见）
- **规则**：方向类判断先 `console.log` 打印关键值确认，不要凭直觉写

### 3. useEffect 依赖数组陷阱
- `userStarted` 加到依赖数组导致 HMR 报错
- 改为 `requestAnimationFrame` 循环检测 `video.readyState` 解决
- **规则**：动态状态变量不要直接进 useEffect 依赖数组，用 ref + rAF 替代

### 4. 手机端 getUserMedia 交互限制
- 自动调用被浏览器隐私策略拦截
- 必须用户手动点击按钮触发
- **规则**：移动端所有需要权限的 API 都要绑定用户手势事件
