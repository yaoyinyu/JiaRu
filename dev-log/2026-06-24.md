## 二、每项任务详情

### 任务 1：全局朝向渲染门控恢复

**问题**: `shouldRenderNails()` 用 4 传感器融合正确检测到了手心/手背，但它的 `decision.render` 返回值被忽略，仅用于设置 UI 朝向指示器。唯一阻止渲染的防线是 `isNailVisible()` 的 `TIP.z - DIP.z > 0.002`，但 MediaPipe z 坐标精度不足，手心朝镜头时 z 差值经常不超阈值。

**根因**: commit 6a9e941 移除了全局 decision.render 门控，改为纯逐指判定。但逐指 z-diff 不够可靠。

**修复**: 在 onResults 回调中添加 `if (decision.render)` 门控：
- 手心（palmScore ≥ 2 或叉积+辅助）→ `render=false` → 跳过 paintNails
- 手背/侧手/模糊态 → `render=true` → 进入 paintNails

**最终两层防御架构**:
```
Layer 1 (全局): shouldRenderNails() — 4 传感器融合 → 手心跳过整帧
Layer 2 (逐指): computeFingerVisibility() / isNailVisible() — 逐指过滤
```

**文件**: `src/components/ArView.tsx:883-890`
**文档**: `docs/global-render-gate-fix.md`

### 任务 2：三信号融合逐指可见性检测

**问题**: 旧 `isNailVisible()` 仅用 `TIP.z - DIP.z > 0.002` 单一信号。MediaPipe z 坐标是单目深度估计，精度远低于 x/y，导致混合态（手偏转时部分指可见）判定不可靠。

**修复**: 三信号融合 + 时序平滑

| 信号 | 方法 | 阈值 | 优势 |
|------|------|:----:|------|
| A | TIP.z - DIP.z（平滑后 z） | 0.002 | 传统 z 差改进 |
| B | TIP.z - PIP.z（印证 A） | 0.003 | PIP 比 DIP 更稳定 |
| C | len2D/len3D 透视缩短比 | 0.65 | 基于 x/y 几何，精度最高 |

**融合策略**:
- 强否定：sigA + sigC 同时否定 → 不可见（保守防误判）
- 多数投票：≥ 2 信号认可 → 可见

**时序平滑**:
- 新增 `visibleSmoothRef[5]`
- EMA(α=0.3) 帧间平滑，阈值 0.5（≈ 2 帧确认）
- 消除单帧噪声导致的闪烁

**文件**: `src/components/ArView.tsx` — computeFingerVisibility() 替代 isNailVisible()
**文档**: `docs/finger-visibility-enhancement.md`

### 任务 6：手势兼容检测 — 手指伸展角（信号 D）

**问题**: 用户比耶/握拳/点赞等非全开手势时，弯折的手指仍被渲染美甲。现有三信号(A/B/C)全部基于手指与摄像头相对朝向，无法区分「伸直」和「弯折」。

**方案**: 新增信号 D — 手指伸展角

```
PIP─→DIP─→TIP  伸直：PIP→DIP 和 DIP→TIP 同方向 → 夹角 ≈ 0°
PIP─→DIP        弯折：DIP→TIP 偏离 → 夹角 > 35°
          ↘ TIP
```

**优势**: 只用 x/y 坐标（MediaPipe 精度最高），不受 z 噪声影响。

**4 信号融合（升级）**:
| 手势 | 可见 | 信号 D 作用 |
|------|------|-------------|
| 🖐️ 五指 | 五指 | 全部 sigD=true → 五指渲染 |
| ✌️ 比耶 | 食指+中指 | 无名指+小指弯折 → sigD=false → 跳过 |
| 🤘 摇滚 | 食指+小指 | 中指+无名指弯折 → sigD=false → 跳过 |
| 👍 点赞 | 拇指 | 拇指 sigD=true，其余弯折 → 仅拇指渲染 |
| 👊 握拳 | 无 | 全部 sigD=false → 全跳过 |
| ☝️ 食指指 | 食指 | 食指 sigD=true，其余弯折 → 仅食指渲染 |
| 🤙 打电话 | 拇指+小指 | 拇指+小指 sigD=true → 仅这两指渲染 |

**变化**: 3→4 信号，强否定=4 全否，多数投票≥2/4

**文件**: `src/components/ArView.tsx` — computeFingerExtensionAngle() + 常量
**文档**: `docs/gesture-compatible-detection.md`

## 八、经验教训（10+2 条）

### 8.1 已有 4 条（从原始笔记迁移）
1. **阈值设计画数轴** — 避免包含关系导致逻辑恒为 true
2. **z方向先实测再写代码** — MediaPipe z 朝镜头为负值（非直觉），先 console.log 确认
3. **useEffect 不放动态 state** — HMR 报错，用 useRef + rAF 替代
4. **移动端权限 API 必须手势触发** — 浏览器隐私策略拦截自动调用

### 8.2 今日新增 8 条
5. **CSS object-cover 需要坐标补偿** — canvas 显示尺寸 ≠ video 原始尺寸，landmarks 坐标需根据裁剪偏移 + 缩放比例变换

6. **rAF 循环中闭包捕获 stale state** — useEffect 内的 rAF/onResults 回调引用 React state 时，闭包捕获的是 effect 运行时的值，不是最新值。解决方案：用 useRef 同步写入，回调从 ref 读取。

7. **前置摄像头镜像行为因设备/OS 而异** — 部分设备对前摄帧做了预镜像，导致 multiHandedness 标签可能与实际手相反。不能自动补偿（无法判断设备是否镜像），提供手动反转开关让用户控制。

8. **方向向量用 TIP-DIP 比 DIP-PIP 更准** — 指甲长在远端指骨（TIP-DIP段），DIP-PIP 在手指弯曲时与指甲方向有偏差。尤其当用户握拳或手指弯曲时，DIP-PIP 方向与指甲实际朝向差异明显。

9. **强否定越宽松漏判越少** — 严格命名「!sigA && !sigC → 隐藏」在混合态场景下漏判严重。逐步放宽到「2否定→3否定→4否定才隐藏」，每次放宽都减少了误判隐藏。

10. **先写文档再写代码** — 按「技术文档→验收标准→实现→验收→标记完成」流程工作，减少返工，验收通过后自然保证质量。

11. **独立信号越多动作兼容性越好** — 信号 A/B/C 都基于手指与摄像头朝向，无法区分「伸直」和「弯折」。新增信号 D（伸展角）后，✌️/👍/👊/☝️ 等手势自动正确。因为伸展角使用了不同的物理原理（骨节夹角 vs 朝向投影），与前三者完全正交。

12. **useRef + useRef 同部避免额外渲染** — handFlip 需要 UI 响应（useState）同时需要闭包安全（useRef）。在 setState 的回调中同步写入 ref，单次操作完成两个目标。
