### 二、本次新增能力

#### 1. 只有通过决策报告才允许继续注册版本

`promote-approved-release.ts` 会先读取：

- `release-decision-report.json`

然后再回读其中的：

- `pipelineReportPath`

并从 pipeline report 里拿到：

- `paths.manifestPath`

只有确认决策状态允许发布后，才会继续调用：

- `scripts/register-model-release.ts`

#### 2. 默认发布规则

- `approve_candidate`：允许发布
- `manual_review`：默认不允许
- `hold_candidate`：禁止发布

#### 3. 支持显式人工放行

对于 `manual_review`，只有在明确传入：

- `--allow-manual-review true`

时，才允许继续注册 candidate。

#### 4. 生成 promotion-report

脚本会输出：

- `promotion-report.json`

其中记录：

- `decisionStatus`
- `candidateVersion`
- `manifestPath`
- `registerSummary`
- 如果被阻止，则记录 `reason` 和 `nextActions`

### 五、当前阶段意义

到这里，Phase 5 的主链已经从“分散工具脚本”基本收束成了完整闭环：

1. 训练发布流水线

### 六、补充进展：批次 / 版本 / 审计追踪索引

今天继续把 Phase 5 往“真实回溯”这侧补了一层，新增：

- `scripts/build-release-trace-index.ts`

这个脚本不是新的训练或发布动作，而是一个追踪索引入口，用来把以下几段信息串起来：

它输出的 `release-trace-index.json` 会重点回答：

当前索引里已经能稳定汇总：

- `batch.sourceGroup`
- `release.finalAuditReportPath`
- `release.finalAuditFailureSummaryPath`
- `release.firstRunRecordPath`
- `decision.status`
- `promotion.registeredVersion`
- `registry.currentVersion`
- `links.sourceGroupToCandidateVersion`

### 九、补充进展：版本 / 批次历史台账清单

今天继续把 Phase 5 从“单次追踪”推进到“长期历史”这一层，新增：

- `scripts/build-release-history-manifest.ts`

这个脚本的定位不是单次报告，而是把多份：

- `release-trace-index.json`

汇总成一份长期可维护的历史台账：

- `release-history-manifest.json`

当前已经支持汇总的核心字段包括：

- `candidateVersion`
- `sourceGroup`
- `importedFileCount`
- `finalAuditStatus`
- `derivedAnnotationFailures`
- `postprocessFailures`
- `decisionStatus`
- `registeredVersion`
- `currentRegistryVersion`

同时还会自动统计：

- `totals.traceIndexes`
- `totals.uniqueCandidateVersions`
- `totals.uniqueSourceGroups`
- `decisionCounts`
- `finalAuditStatusCounts`

这意味着后面如果我们已经积累了多版 candidate、多次 seed batch、多轮 audit，就可以不止看某一版，而是能从一份 manifest 里直接看整个历史演进。

### 十一、补充进展：release trace 自动登记入口

今天继续把历史台账这条线补成“可自动追加”的入口，新增：

- `scripts/register-release-trace-index.ts`

这条脚本的定位是：

- 输入一份新的 `release-trace-index.json`
- 自动读取已有 `release-history-manifest.json`
- 把新的 trace index 合并进去
- 再调用 `build-release-history-manifest.ts` 重建最新台账

这样后面每新增一版 release，就不用再手工把所有旧的 `--trace-index` 重新拼一遍了。

当前这条自动登记入口会输出：

- `trace-registration-report.json`

其中会记录：

- `traceIndexPath`
- `historyManifestPath`
- `traceIndexCount`
- `includedTraceIndexes`

### 十三、补充进展：reviewed batch import 自动生成初始 release trace 草稿

今天继续把 Phase 5 往“数据一入库就开始挂 release 链路”这个方向推进，新增：

- `scripts/build-initial-release-trace-draft.ts`

并且把它接进了：

- `model/training/run-reviewed-batch-import-pipeline.ts`

现在 reviewed batch import pipeline 在 `import-reviewed-batch` 完成后，会自动生成：

- `release-trace-draft.json`

这个草稿的定位不是完整 release trace，而是把最基础的 batch 侧上下文先固定下来，包括：

- `batch.rootDir`
- `batch.sourceGroup`
- `batch.datasetRoot`
- `batch.reviewedImportReportPath`
- `batch.importedFileCount`

同时把 release / decision / promotion / registry 先保留为空位，等后续真实 release 发生时再继续补齐。

这样后面就不需要等到 release pipeline 全部跑完，才开始建立 trace；而是在 batch 正式进入 dataset 的那一刻，就已经有了第一版 trace 草稿。

### 十五、补充进展：trace 草稿升级为正式 release trace

今天继续把 trace 链路往前收口，补了一层“草稿升级正式 trace”的能力：

- `scripts/build-release-trace-index.ts` 现在支持读取 `release-trace-draft.json`

新增能力点：

- 新增参数 `--release-trace-draft`
- 当存在 draft 时，会优先复用其中的 batch 上下文：
  - `rootDir`
  - `sourceGroup`
  - `datasetRoot`
  - `reviewedImportReportPath`
  - `importedFileCount`
- 然后再用真实的 training release pipeline / final audit / decision / promotion / registry 信息去补齐正式 trace

这意味着：

- batch 刚入库时可以先生成 `release-trace-draft.json`
- 等后面真实 release 发生时，不需要从头再拼 batch 侧信息
- 直接把 draft 升级成完整的 `release-trace-index.json`
