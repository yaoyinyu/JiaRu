## 一、方案 A 数学推导审核 — ❌ 原方案有重大错误

### 1.1 原方案 A（叉积法向量）的核心逻辑

原方案第 3.1.2 节声称：
> 手掌法向量指向镜头 → `normal.z < 0` → 手背朝镜头

### 1.2 数学验证

3D 叉积 `n = v1 × v2`：
```
n.x = v1.y * v2.z - v1.z * v2.y
n.y = v1.z * v2.x - v1.x * v2.z
n.z = v1.x * v2.y - v1.y * v2.x   ← 只用了 x、y 分量
```

**原方案的分析（错误）**：
> `n.z` 公式不包含 z 坐标信息，无法区分手心/手背

**这个分析本身是错误的**。原因：

`n.z = v1.x * v2.y - v1.y * v2.x` 虽然只用 x、y，但它捕捉的是 **v1 和 v2 在图像平面上的排列方向（手性/chirality）**。手心朝镜头 vs 手背朝镜头时，P0→P5→P17 三点在图像平面上的排列方向是**镜像相反的**，所以 `n.z` 的符号确实会翻转。

**但新的致命问题**：`n.z` 的符号**同时依赖左右手**。
- 右手手背朝镜头：`n.z` 符号 = A
- 左手手背朝镜头：`n.z` 符号 = -A（因为左右手在图像中是镜像的）
- 如果不区分左右手，无法用 `n.z` 符号统一判断"手背朝镜头"

### 1.3 修正方案 A（深度差法）

原方案文档第 4 节已提出修正：比较手掌中心 z 与指关节 z 的深度差。

```
palmZ = avg(lm[0].z, lm[5].z, lm[9].z, lm[17].z)
knuckleZ = avg(lm[2].z, lm[6].z, lm[10].z, lm[14].z, lm[18].z)
depthDiff = palmZ - knuckleZ
```

**数学验证**：
- z 负 = 靠近镜头
- 手背朝镜头：PIP 关节凸起 → PIP.z 更负 → `knuckleZ < palmZ` → `depthDiff > 0` ✅
- 手心朝镜头：手掌中心凸起 → 手掌 z 更负 → `depthDiff < 0` ✅
- **对左右手都成立**（不依赖 handedness）✅

### 1.4 结论

| 方案 | 原方案 | 修正后 |
|------|--------|--------|
| 叉积法向量 n.z | ❌ 不依赖 handedness 时无法区分左右手 | 弃用 |
| 深度差法 | ✅ 不依赖 handedness | **采用为主方案** |

## 二、方案 B 审核 — ✅ 逻辑正确但需调整

### 2.1 原方案 B 逻辑

```
dz = TIP.z - PIP.z
dz > 0 → PIP 更靠近镜头 → 手背投票
```

**验证**：
- 手背朝镜头：PIP 凸起 → PIP.z < TIP.z → `dz = TIP.z - PIP.z > 0` ✅
- 手心朝镜头：TIP 凸起 → TIP.z < PIP.z → `dz < 0` ✅
- 对左右手都成立 ✅

### 2.2 拇指问题

拇指的 PIP 是关键点 2（THUMB_MCP，实际上是掌指关节），TIP 是 4。
- 拇指解剖结构与其他指不同（只有 1 个指间关节）
- 拇指的 z 值可能不稳定
- **建议**：拇指投票权重降低或排除

### 2.3 结论

方案 B 逻辑正确，建议排除拇指（只用 4 指投票），阈值需实测调优。

## 三、z 坐标可靠性风险评估 — ⚠️ 中等风险

### 3.1 z 坐标特性

MediaPipe Hands 的 z 坐标：
- **不是绝对深度**，是"相对手腕的深度偏移"
- 基于**单目视觉**（monocular）的深度估计，不是真实 3D 测量
- 精度低于 x、y（x、y 是 2D 像素归一化，z 是模型推断的相对深度）
- 帧间噪声较大

### 3.2 z 坐标在"手心/手背区分"上的可靠性

**有利因素**：
- 我们只需要**符号**（正/负），不需要精确数值
- 手背 vs 手心的深度差是**结构性差异**（关节凸起 vs 手掌凸起），不是随机噪声
- 4 指投票可以平均掉单指噪声

**不利因素**：
- 远距离时手部像素少，z 估计更不稳定
- 手指弯曲时 PIP 和 TIP 的 z 关系可能变化
- 侧手时所有 z 差值都很小，容易误判

### 3.3 风险缓解

- EMA 平滑（α=0.3）减少帧间抖动 ✅
- 阈值 + 滞回避免临界值跳变 ✅
- 安全降级（模糊态不渲染）✅
- **新增建议**：当手部在画面中占比过小（<15% 面积）时，降低置信度

## 四、与现有代码集成审核 — ⚠️ 发现 3 个集成问题

### 4.1 问题 1：onResults 回调中的渲染流程

**现有流程**（line 581-600）：
```typescript
handsInst.onResults((res: HandResults) => {
  ctx.clearRect(0, 0, cvs.width, cvs.height);
  if (res.multiHandLandmarks?.length) {
    for (let h = 0; h < res.multiHandLandmarks.length; h++) {
      paintNails(ctx, res.multiHandLandmarks[h], ...);
    }
  }
});
```

**集成方案**：在 `paintNails()` 调用前判断是否应该渲染。

**问题**：`paintNails()` 内部对每根手指独立处理，朝向判断应该在哪一层做？
- 选项 A：在 `onResults` 回调中，对整只手做一次判断 → 决定是否调用 `paintNails()`
- 选项 B：在 `paintNails()` 内部，对每根手指独立判断

**正确选择**：选项 A。因为手心/手背是**整只手**的属性，不是单指属性。同一只手不可能有些指是手背、有些指是手心。

### 4.2 问题 2：smoothRef 只存储 5 指数据

现有 `smoothRef` 只存储 tip/dip/pip 的平滑值，**不包含手掌关键点（0, 5, 9, 17）的平滑值**。

如果朝向判断需要手掌关键点的 z 值，需要：
- 新增一个 `palmSmoothRef` 存储手掌关键点的 EMA 平滑 z 值
- 或者直接使用原始 z 值（不推荐，会增加噪声）

**建议**：新增 `palmZSmoothRef`（单个数值，EMA 平滑 `depthDiff`）。

### 4.3 问题 3：现有 paintNails 内部的 z 使用

`paintNails()` 内部已有 z 深度缩放逻辑（line 390）：
```typescript
const zScale = Math.max(0.7, Math.min(1.5, 1 - s.tip.z * 0.6));
```

这个 `zScale` 影响指甲大小（深度感知缩放）。如果手心朝镜头时不渲染，这个逻辑不受影响。但如果朝向判断错误，会导致：
- 手心时渲染了美甲 → 视觉错误（主要问题）
- 手背时不渲染 → 用户看不到美甲（体验问题）

两者都是**体验降级**，不是**数据损坏**或**安全问题**。风险可控。

### 4.4 结论

集成问题均可通过合理设计解决，不会引入新的功能性问题。

| 操作 | 复杂度 | 预估耗时 |

## 七、边界情况审核 — ⚠️ 发现 4 个未覆盖场景

### 7.1 场景 1：手指弯曲

用户握拳或半握拳时：
- PIP 关节凸起更明显（手背）→ `depthDiff` 应该更大 → 判断更准确 ✅
- 但 `paintNails()` 中的 `rawLen < 5` 会跳过该指 → 不渲染该指甲
- 朝向判断不受影响 ✅

### 7.2 场景 2：手部部分出画面

手部部分出画面时，MediaPipe 可能仍返回关键点（x、y 超出 [0,1]）。
- z 值可能不可靠
- **建议**：如果任何手掌关键点（0,5,9,17）的 x、y 超出 [0.1, 0.9] 范围，降低置信度

### 7.3 场景 3：快速翻转

用户快速翻转手掌（<100ms）：
- EMA α=0.3 的平滑延迟约 3-5 帧（100-167ms at 30fps）
- 翻转期间可能短暂渲染错误
- **可接受**：这是过渡态，用户不会注意到 100ms 的错误

### 7.4 场景 4：多手场景

当前 `maxNumHands: 1`，但如果未来改为多手：
- 朝向判断对每只手独立执行
- 现有 `for (let h = 0; h < res.multiHandLandmarks.length; h++)` 循环已支持
- **无新增问题** ✅

### 7.5 未覆盖场景汇总

| 场景 | 影响 | 缓解 |
|------|------|------|
| 手指出画面 | z 不可靠 | 检查 x、y 范围，降低置信度 |
| 极近距离 | 手掌占满画面，z 估计可能失真 | 检测手部占比 >80% 时降低置信度 |
| 手部快速运动 | EMA 平滑延迟 | 可接受（<167ms 过渡态） |
| 光照极暗 | MediaPipe 检测置信度下降 | `minDetectionConfidence: 0.5` 已有保护 |

## 九、审核结论

### 9.1 方案可行性

**目标**：仅在手背朝镜头时渲染美甲纹理。

**结论**：✅ **可行，但需修正方案 A**

| 修正项 | 原方案 | 修正后 |
|--------|--------|--------|
| 主方案 | 叉积法向量 n.z | 深度差法（palmZ - knuckleZ） |
| 辅助方案 | 5 指投票 | 4 指投票（排除拇指） |
| 集成层 | 在 paintNails 内部判断 | 在 onResults 回调中判断整只手 |
| 平滑 | smoothRef 扩展 | 新增 palmZSmoothRef |

### 9.2 新问题风险评估

| 风险 | 概率 | 影响 | 可控性 |
|------|:----:|:----:|:----:|
| z 噪声导致误判 | 中 | 低（体验降级） | ✅ EMA + 阈值 + 安全降级 |
| 叉积法误用（已修正） | — | — | ✅ 已弃用 |
| UI 遮挡 | 低 | 低 | ✅ 位置调整 |
| 性能下降 | 极低 | 极低 | ✅ <0.03ms/帧 |
| 现有功能受影响 | 极低 | 低 | ✅ 朝向判断在渲染前做，不影响 paintNails 内部逻辑 |

### 9.3 最终建议

1. **弃用叉积法向量方案**，改用深度差法作为主方案
2. **排除拇指投票**，只用 4 指投票
3. **在 `onResults` 回调中做整手判断**，不在 `paintNails` 内部做
4. **新增 `palmZSmoothRef`**，不修改现有 `smoothRef`
5. **朝向指示器放在 ready 状态的底部状态栏**，不修改诊断面板逻辑
6. **阈值需实测调优**，初始值 `depthDiff > 0.003` 作为手背判断标准
7. **修正技术方案文档**，更新为深度差法
